# 图书馆座位预约系统 — 需求分析与MVP确认

## 1. 目标用户与核心场景

### 1.1 目标用户画像

| 用户类型 | 身份描述 | 典型行为 |
|----------|----------|----------|
| 学生用户 A：自习党 | 大三/大四学生，每天固定时间到图书馆自习 | 提前一天查座位、预约常坐的区域、偶尔取消 |
| 学生用户 B：临时党 | 课间或周末想去图书馆，不固定时间 | 临时打开系统看看有没有空位，有就约、没有就走 |
| 管理员 C：值班老师 | 图书馆工作人员，每天查看预约情况 | 早上登录后台看今日预约量，偶尔处理座位停用/启用 |

### 1.2 核心使用场景

| 场景编号 | 场景名称 | 场景描述 |
|----------|----------|----------|
| SC-01 | 考前预约高峰 | 考试周前三天，学生集中预约自习座位，系统需要正确展示座位余量 |
| SC-02 | 日常浏览预约 | 普通上课日，学生课间浏览座位列表，预约下午时段的座位 |
| SC-03 | 临时取消 | 学生临时有事不去图书馆，在"我的预约"中取消，释放座位 |
| SC-04 | 管理员巡检 | 管理员每天登录后台，查看当日预约数量和各区域分布 |
| SC-05 | 座位维护 | 某座位设备故障，管理员在后台将其标记为"停用"，用户端不再显示为可约 |

## 2. 用户最关心的问题

| 序号 | 用户关心的问题 | 对应系统能力要求 |
|------|---------------|------------------|
| Q1 | 现在还有没有空位？ | 座位列表按区域展示，清楚标注"可预约"或"已满" |
| Q2 | 我想约的那个座位具体在哪儿？有什么设施？ | 座位详情页展示位置描述和设施信息 |
| Q3 | 我能不能约到明天下午？ | 选择日期和时段，系统检查冲突后确认 |
| Q4 | 我之前约了哪天哪个座位？ | 我的预约页列出全部预约记录 |
| Q5 | 不去了怎么取消？ | 我的预约页提供取消按钮，点击即生效 |
| Q6 | 管理员怎么知道今天来了多少人？ | 后台统计页展示今日预约总数和区域分布 |

## 3. 候选功能池

基于立项单范围 + 校园图书馆场景补充，列出全部候选功能：

| 编号 | 功能 | 所属端 | 说明 |
|------|------|--------|------|
| F01 | 用户首页 | 用户端 | 展示图书馆信息 + 快捷入口 |
| F02 | 体验账号切换 | 用户端 | 预设几个学生账号，下拉框切换 |
| F03 | 座位列表浏览 | 用户端 | 按区域分组展示座位，显示可约/已满状态 |
| F04 | 座位详情查看 | 用户端 | 单个座位的详细信息（位置、设施、当前状态） |
| F05 | 预约提交（选日期+时段） | 用户端 | 从座位详情进入，选日期和时段提交预约 |
| F06 | 预约冲突检查 | 用户端/后端 | 同一座位同一日期同一时段不可重复预约 |
| F07 | 我的预约列表 | 用户端 | 展示当前用户全部预约，区分"已预约""已取消"状态 |
| F08 | 取消预约 | 用户端 | 在"我的预约"中取消未到期的预约 |
| F09 | 管理员登录 | 管理端 | 账号密码登录，Session/Cookie 维持会话 |
| F10 | 预约管理（筛选+查看） | 管理端 | 按日期、状态、区域筛选预约记录 |
| F11 | 座位管理（增/改/停用） | 管理端 | 添加新座位、修改座位信息、停用/启用座位 |
| F12 | 统计页（今日数据） | 管理端 | 今日预约总数、各区域预约分布 |
| F13 | 预约记录导出 | 管理端 | 将预约记录导出为 CSV/Excel |
| F14 | 用户反馈/报修 | 用户端 | 学生对座位故障进行反馈 |
| F15 | 预约提醒（浏览器通知） | 用户端 | 预约时间快到时浏览器弹通知 |
| F16 | 座位收藏 | 用户端 | 收藏常用座位，快速预约 |

| | F03 | 座位列表浏览 | 用户端核心能力 |

## 5. MVP 范围确认

### 5.1 MVP 选定范围

**P0 + P1 全部纳入 MVP，P2 和 P3 全部排除。**

MVP 共包含 12 项功能（F01-F12），对应立项单中的 9 个页面 + 体验账号切换 + 预约冲突检查。

### 5.2 MVP 验证方式

| 验证项 | 验证方法 |
|--------|----------|
| 用户能否完成一次完整的预约？ | 切换账号 → 浏览座位 → 查看详情 → 提交预约 → 在"我的预约"中确认 |
| 预约冲突是否能正确拦截？ | 对同一座位/日期/时段尝试两次预约，第二次应被拦截 |
| 管理员能否管理座位和预约？ | 登录后台 → 查看预约 → 筛选 → 添加座位 → 停用座位 |
| 统计是否正确？ | 手动创建几条预约，统计页数字应一致 |

## 6. 本期明确不做的功能

| 编号 | 功能 | 不做的原因 |
|------|------|------------|
| F13 | 预约记录导出 | P2 优先级，课堂项目工作量不划算 |
| F14 | 用户反馈/报修 | 超出立项单范围 |
| F15 | 预约提醒（浏览器通知） | 需要浏览器通知权限，复杂度高 |
| F16 | 座位收藏 | P2 优先级，主链路不依赖 |
| — | 支付 | 不涉及费用 |
| — | 扫码签到/签退 | 需要硬件+移动端，超出范围 |
| — | 短信/微信推送 | 需要第三方服务 |
| — | 用户注册/邮箱验证 | 本期用体验账号替代 |
| — | 多馆区 | 单图书馆即可 |
| — | 实时 Floor Map | 开发量太大 |

## 7. 需求评审结论

### 7.1 保留项（进入 MVP）

全部 P0 + P1 功能（F01-F12），共 12 项：

- 用户端：用户首页、体验账号切换、座位列表、座位详情、预约提交、预约冲突检查、我的预约、取消预约
- 管理端：管理员登录、预约管理、座位管理、统计页

### 7.2 删减项（不进入本期）

P2 功能（F13 预约记录导出、F16 座位收藏）— 本期不开发。

### 7.3 暂缓项（留待将来）

P3 功能（F14 用户反馈/报修、F15 预约提醒）— 本期不开发，如果后续有精力可考虑。

## 8. 进入 PRD Lite 前的最终口径

以下 6 条是本阶段确认的硬边界，进入 PRD Lite 后不得再扩大：

1. **9 个页面**：用户端 5 页 + 管理端 4 页，不可增页。
2. **12 项 MVP 功能**：P0+P1 全部，P2/P3 全部排除。
3. **单图书馆、单日期单时段单座位不可重复预约**。
4. **用户端用体验账号下拉框切换，管理端用 Session/Cookie 登录**。
5. **不做支付、不做扫码、不做推送、不做注册、不做 Floor Map**。
6. **最终交付物为可运行的 ASP.NET Core MVC 项目源码 + 完整文档 + GitHub/Gitee 仓库链接**。
