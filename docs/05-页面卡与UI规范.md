## 页面卡 P01：用户首页

| 维度 | 内容 |
|------|------|
| **页面名称** | 用户首页 |
| **页面角色** | 学生用户 |
| **页面目标** | 作为系统入口，让用户快速导航到"找座位"或"我的预约" |
| **页面入口** | 直接访问 `/` |

| **最重要的信息** | 图书馆名称、今日开放时间、当前体验账号名称 |

| **页面角色** | 学生用户 |

## 页面卡 P04：预约提交页

| 维度 | 内容 |
|------|------|
| **页面名称** | 预约提交页 |
| **页面角色** | 学生用户 |
| **页面目标** | 确认座位、选择日期和时段，完成预约提交 |
| **页面入口** | 从 P03 点击"去预约"，路由 `/Reservation/Create?seatId={id}` |
| **页面出口** | 提交成功 → P05 我的预约页；提交失败 → 本页（显示错误）；点击返回 → P03 |
| **最重要的信息** | 已选座位信息（确认区）、日期和时段选择器 |
| **最重要的操作** | "提交预约"按钮（主操作） |
| **成功状态** | 提交成功后自动跳转 P05，新预约记录出现在列表顶端 |
| **空状态** | 该座位今天所有时段已被约满：日期选择器切换后时段全红 + 提示"该日所有时段已满" |
| **异常提示** | 冲突："该时段已被他人预约，请选择其他时段"；座位停用："该座位已停用，无法预约"；未选时段："请选择预约时段" |
| **布局要求** | 手机竖屏；顶部返回栏 + 已选座位确认卡片（座号、区域）+ 日期选择器（横向滑动或下拉）+ 时段选择列表（每个时段一行，绿色可选/红色已满）+ 底部固定"提交预约"按钮 |
| **数据关键点** | 提交时后端执行原子性冲突检查（同一座位+日期+时段是否已有 Active 预约），必须在 INSERT 前检查 |

## 页面卡 P05：我的预约页

| 维度 | 内容 |
|------|------|
| **页面名称** | 我的预约页 |
| **页面角色** | 学生用户 |
| **页面目标** | 展示当前用户全部预约，支持取消操作 |
| **页面入口** | 从 P01 点击"我的预约"或从 P04 提交成功后跳转，路由 `/Reservation/My` |
| **页面出口** | 取消后本页刷新；点击返回 → P01 |
| **最重要的信息** | 每条预约的：座位号、区域、日期、时段、状态标签 |
| **最重要的操作** | "取消"按钮（仅对 Active 状态的预约生效） |
| **成功状态** | 列表正常展示，已预约（Active）以绿色标签显示，已取消（Cancelled）以灰色标签显示 |
| **空状态** | 当前用户无任何预约记录：显示空状态插图 + "你还没有预约记录" + "去找座位"按钮（跳转 P02） |
| **异常提示** | 取消失败："取消失败，请稍后重试"；数据加载失败："加载失败，请刷新重试" |
| **布局要求** | 手机竖屏；顶部返回栏 + 预约记录列表（卡片式，每条包含座号/区域/日期/时段/状态标签/取消按钮） |
| **数据关键点** | 按当前体验账号 UserId 查询 Reservation（含 Seat、SeatArea、TimeSlot 关联数据）；取消操作 UPDATE Reservation.Status = 'Cancelled' |

## 3. 视觉方向说明

**主方向：安静校园学习感 + 现代微服务感**

- 用户端：以"安静学习"为关键词，通过柔和绿色系、留白呼吸感、卡片式布局营造"图书馆的宁静"。拒绝花哨渐变、拒绝高饱和撞色。
- 管理端：以"现代后台"为关键词，通过左侧深色导航、清爽数据表格、简洁操作按钮营造"高效工作台"感觉。

---

## 4. 页面氛围关键词与禁用风格

### 关键词
- 用户端：安静、温暖、可信、呼吸感、学院气
- 管理端：高效、干净、秩序感、专业、克制

### 禁用风格
- ❌ 高饱和霓虹色
- ❌ 大面积渐变背景
- ❌ 复杂动画/过度动效
- ❌ emoji 作为 UI 元素
- ❌ 拟物化阴影、3D 效果
- ❌ 花哨字体（手写体、艺术字等）
- ❌ 暗黑模式（本期不做）
- ❌ 弹窗叠加弹窗

## 6. 响应式原则

| 断点 | 宽度 | 适配策略 |
|------|------|----------|
| 手机 | < 768px | 用户端主战场：单列布局，全宽卡片，底部固定按钮，顶部返回导航 |
| 平板 | 768px-1024px | 用户端可双列卡片；管理端保持侧边栏+内容区 |
| 桌面 | > 1024px | 管理端主战场：完整侧边栏 + 宽表格 + 数据卡片行；用户端居中最大宽度 480px 模拟手机 |

**核心原则**：用户端永远不超过 480px 内容区宽度（手机优先），管理端充分利用桌面宽度。

## 7. 原型三层次说明

| 层次 | 定义 | 交付物 |
|------|------|--------|
| **低保真** | 灰度线框图，只展示布局结构和内容优先级，不加颜色和真实图片 | 手绘风格或 Balsamiq 风格线框图 |
| **中保真** | 带真实色彩、字体、间距、组件的可点击静态 HTML 页面，使用 Bootstrap + 自定义 CSS | HTML + CSS 静态页面，可点击跳转 |
| **高保真** | 中保真基础上接入后端数据和真实交互逻辑 | 不归原型阶段，属于正式开发产出 |

## 8. 哪些页面先做低保真，哪些直接做中保真

| 策略 | 页面 | 原因 |
|------|------|------|
| **直接中保真** | P01 用户首页、P02 座位列表页、P03 座位详情页、P04 预约提交页、P05 我的预约页 | 用户端核心链路 5 页，直接影响用户体验感受，必须直接看到视觉效果 |
| **直接中保真** | P07 预约管理页、P08 座位管理页 | 管理端核心操作页，需验证表格/表单/弹窗的可用性 |
| **直接中保真** | P06 管理员登录页 | 简单页面，中保真成本低 |
| **直接中保真** | P09 统计页 | 数据卡片和表格布局需要真实视觉验证 |

**结论**：9 页全部直接出中保真静态 HTML，不单独做低保真阶段。原因：使用 Bootstrap 框架，中保真成本已经很低，直接出视觉效果更有利于评审。

## 9. 原型评审时要重点看什么

| 评审重点 | 具体检查项 |
|----------|------------|
| 核心流程完整性 | 能否从 P01 → P02 → P03 → P04 → P05 走通（页面间可点击跳转） |
| 手机端适配 | P01-P05 在 375px 宽度下是否像真实手机页面（非拉伸的桌面页） |
| 状态展示 | 是否能看到"可约"和"已满"两种座位的视觉差异；是否能看到空状态页面 |
| 视觉统一性 | 9 页色彩、圆角、间距、按钮风格是否一致（不能每页像不同项目） |
| 表单可用性 | 日期选择、时段选择、登录表单是否真实可用（有输入框、有按钮、有提示） |
| 管理端布局 | 左侧导航 + 内容区是否清晰；表格是否有表头/数据行/操作列 |
| 统计页完整度 | 是否至少有数据卡片 + 区域分布表格 |
