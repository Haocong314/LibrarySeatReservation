# 08-数据库设计-审计

## A. 总体判断

**通过。** 数据库设计文档完整覆盖了从实体提取、ER 关系到表结构、从字段定义到索引策略、从状态流转到冲突约束的全套设计。5 个实体（Users / Admins / SeatAreas / Seats / Reservations）与前序文档（03-PRD-Lite / 04-页面树 / 07-系统设计说明）的命名和范围一致，未擅自将前序已有更细设计删平为固定 6 实体。表关系清晰（3 条 1:N），预约冲突规则通过过滤唯一索引 + 应用层双重保障。设计体量适中（5 表、每表 ≤ 8 字段），非常适合 SQL Server LocalDB + EF Core + 单人课堂项目。

## B. 发现的问题清单

| 编号 | 问题 | 严重程度 |
|------|------|----------|
| B-01 | `Admins` 表存储 `PasswordHash` 为 SHA256，注释中写"不强制 BCrypt"。SHA256 缺乏盐值（salt），若被拖库则易被彩虹表攻击。但对课堂项目来说可接受 | 低 |
| B-02 | 种子数据中 Seats 表有 7 条记录，但 `Reservations` 表没有种子数据。管理端 P07/P09 首次打开时将显示空表/零统计，建议在 Seed 方法中补充 1-2 条示例预约记录，方便演示 | 低 |
| B-03 | 索引建议中的过滤唯一索引 `CREATE UNIQUE INDEX ... WHERE Status = '已预约'` 依赖 SQL Server 的过滤索引特性，LocalDB 支持此特性，确认无误 | 信息 |

## D. 建议优化项

| 编号 | 建议 | 说明 |
|------|------|------|
| D-01 | 在 Seed 数据中补充 1-2 条示例预约记录（如用户 2024001 在今日预约了 A-01 的 08:00-10:00），让管理端 P07/P09 首次打开时即有数据可展示 | 提升演示效果 |
| D-02 | `Admins.PasswordHash` 在实现时建议至少加随机 salt（格式：`{salt}:{hash}`），即使仍是 SHA256 也能防彩虹表 | 最小安全提升，代码实现量极小 |
