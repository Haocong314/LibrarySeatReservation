# 图书馆座位预约系统 — 关键链路详细设计

## 1. 关键链路选择说明

前序文档（04-页面树与业务流程 / 07-系统设计说明 / 08-数据库设计）已明确两条核心主链路：

| 链路 | 类型 | 路径 | 说明 |
|------|------|------|------|
| 链路一 | 用户预约主链路 | P01 → P02 → P03 → P04 → P05 | 覆盖用户从浏览到成功预约的完整闭环 |
| 链路二 | 管理员取消预约链路 | P06 → P07 → 取消操作 | 覆盖管理员登录后查询并强制取消某条预约 |

链路一是最核心的业务价值路径，涉及 5 个页面、3 个 Controller、2 个 Service、3 张表。链路二是管理端最关键的干预动作，涉及预约状态从"已预约"回流至"已取消"。

## 2. 链路一：用户预约主链路

### 2.1 页面与路由

| 步骤 | 页面 | HTTP 方法 | 路由 | 用户动作 |
|------|------|-----------|------|----------|
| 1 | P01 用户首页 | GET | `/` 或 `/Home/Index` | 点击"找座位" |
| 2 | P02 座位列表页 | GET | `/Seat/List?areaId=1` | 浏览座位卡片，点击某个座位 |
| 3 | P03 座位详情页 | GET | `/Seat/Detail/3` | 查看座位信息 + 今日时段占用，点击"去预约" |
| 4 | P04 预约提交页 | GET | `/Reservation/Create?seatId=3` | 确认日期、选择时段，点击"提交预约" |
| 5 | P04→P05 | POST | `/Reservation/Create` | 后端处理后自动跳转 |
| 6 | P05 我的预约页 | GET | `/Reservation/MyReservations` | 查看刚提交的预约记录 |

### 2.2 Controller 动作（ReservationController）

**GET `/Reservation/Create?seatId=3`** → `Create(int seatId)`

```csharp
[HttpGet]
public async Task<IActionResult> Create(int seatId)
{
    // 1. 从 QueryString 或隐藏字段获取当前体验用户 ID
    int currentUserId = GetCurrentUserId();

    // 2. 调 Service 获取座位信息 + 今日已占时段列表
    var vm = await _seatService.GetSeatForReservationAsync(seatId,
        DateOnly.FromDateTime(DateTime.Today));

    // 3. 若座位不存在或已停用，跳回 P02
    if (vm == null)
        return RedirectToAction("List", "Seat");

    // 4. 传入 View
    return View(vm);
}
```

**POST `/Reservation/Create`** → `Create(ReservationCreateViewModel vm)`

```csharp
[HttpPost]
[ValidateAntiForgeryToken]
public async Task<IActionResult> Create(ReservationCreateViewModel vm)
{
    int currentUserId = GetCurrentUserId();

    if (!ModelState.IsValid)
        return View(vm);

    // 调 Service 执行预约（含冲突检查）
    var result = await _reservationService.CreateReservationAsync(
        currentUserId, vm.SeatId, vm.ReservationDate, vm.TimeSlot);

    if (result.Success)
        return RedirectToAction("MyReservations");
    else
    {
        ModelState.AddModelError("", result.ErrorMessage);
        return View(vm); // 停留在 P04，显示错误（如"该时段已被预约"）
    }
}
```

### 2.3 Service 职责（ReservationService）

**`CreateReservationAsync(int userId, int seatId, DateOnly date, string timeSlot)`**

```
入口参数：userId, seatId, date, timeSlot

步骤 1 — 座位存在性与可用性检查：
  - 查 Seats 表：WHERE Id == seatId
  - 若不存在 → 返回失败 "座位不存在"
  - 若 Status == "停用" → 返回失败 "该座位当前不可预约"

步骤 2 — 预约冲突检查（核心）：
  - 查 Reservations 表：
    WHERE SeatId == seatId
      AND ReservationDate == date
      AND TimeSlot == timeSlot
      AND Status == "已预约"
  - 若存在记录 → 返回失败 "该时段已被预约，请选择其他时段"

步骤 3 — 写入预约（在同一事务内）：
  - INSERT INTO Reservations (UserId, SeatId, ReservationDate, TimeSlot, Status, CreatedAt)
    VALUES (userId, seatId, date, timeSlot, "已预约", DateTime.UtcNow)
  - 数据库唯一索引 IX_Reservations_Conflict 作为最后防线：
    若并发插入导致冲突，EF Core 抛出 DbUpdateException

步骤 4 — 返回成功：
  - 返回 CreateReservationResult { Success = true }
```

> **事务说明**：步骤 2 的查询和步骤 3 的插入必须在同一数据库事务内完成，防止 TOCTOU（Time-of-Check-Time-of-Use）问题。EF Core 中通过 `await _context.Database.BeginTransactionAsync()` 包裹步骤 2+3。

### 2.4 DataAccess 动作（EF Core）

| 步骤 | EF Core 操作 | 对应 SQL | 表 |
|------|-------------|----------|-----|
| 查座位 | `_context.Seats.FindAsync(seatId)` | `SELECT * FROM Seats WHERE Id = @seatId` | Seats |
| 冲突检查 | `_context.Reservations.Where(r => r.SeatId == seatId && r.ReservationDate == date && r.TimeSlot == timeSlot && r.Status == "已预约").AnyAsync()` | `SELECT TOP 1 1 FROM Reservations WHERE ...` | Reservations |
| 插入预约 | `_context.Reservations.Add(entity); await _context.SaveChangesAsync();` | `INSERT INTO Reservations (...) VALUES (...)` | Reservations |

### 2.5 数据库变化

| 操作 | 表 | 影响 |
|------|-----|------|
| 查询座位 | Seats | 只读，无变化 |
| 冲突检查 | Reservations | 只读，无变化 |
| 写入新预约 | Reservations | **新增一行**：Status = '已预约'，CreatedAt = 当前时间 |

### 2.6 状态流转

```
初始状态：无预约记录

[用户提交预约] ──→ Reservations.Status = "已预约"
                          │
                          ├── [用户取消] ──→ Status = "已取消", CancelledAt = 当前时间
                          │
                          └── [管理员取消] ──→ Status = "已取消", CancelledAt = 当前时间
```

> 链路一覆盖"无 → 已预约"这一段。取消操作由链路二或用户自行在 P05 触发。

### 2.7 异常与边界情况

| 边界场景 | 检测层 | 响应 |
|----------|--------|------|
| 座位 ID 不存在 | Service 步骤 1 | 重定向到 P02，提示"座位不存在" |
| 座位已停用 | Service 步骤 1 | 停留在 P04，显示"该座位当前不可预约" |
| 时段已被他人预约 | Service 步骤 2 | 停留在 P04，显示"该时段已被预约" |
| 用户选择过去的日期 | Controller 层 ModelState 或前端 JS | 停留在 P04，显示"请选择今天或之后的日期" |
| 用户未选择时段直接提交 | ASP.NET Core ModelState | 停留在 P04，"请选择时段"红色提示 |
| 并发冲突（两人同时预约同一时段） | 数据库唯一索引 | EF Core 抛出 DbUpdateException，Service 捕获后返回"预约失败，该时段刚刚被抢走" |
| 数据库连接失败 | EF Core 异常 | 全局异常中间件捕获，返回友好错误页 |

## 3. 链路二：管理员取消预约链路

### 3.1 页面与路由

| 步骤 | 页面 | HTTP 方法 | 路由 | 管理员动作 |
|------|------|-----------|------|-----------|
| 1 | P06 管理员登录页 | GET | `/Admin/Login` | 输入账号密码 → 点击登录 |
| 2 | P06 → P07 | POST | `/Admin/Login` | 登录成功 → 跳转 |
| 3 | P07 预约管理页 | GET | `/AdminManage/Reservations?date=2026-07-10&status=已预约` | 筛选查看预约列表 |
| 4 | P07 | POST | `/AdminManage/CancelReservation/5` | 点击某行的"取消预约"按钮 → 确认 |

### 3.2 Controller 动作

**POST `/Admin/Login`** → `AdminController.Login(AdminLoginViewModel vm)`

```csharp
[HttpPost]
[ValidateAntiForgeryToken]
public async Task<IActionResult> Login(AdminLoginViewModel vm)
{
    if (!ModelState.IsValid) return View(vm);

    var admin = await _adminService.ValidateAsync(vm.Username, vm.Password);
    if (admin == null)
    {
        ModelState.AddModelError("", "用户名或密码错误");
        return View(vm);
    }

    HttpContext.Session.SetString("AdminId", admin.Id.ToString());
    return RedirectToAction("Reservations", "AdminManage");
}
```

**POST `/AdminManage/CancelReservation/5`** → `AdminManageController.CancelReservation(int id)`

```csharp
[HttpPost]
[ValidateAntiForgeryToken]
public async Task<IActionResult> CancelReservation(int id)
{
    // 1. Session 鉴权
    if (HttpContext.Session.GetString("AdminId") == null)
        return RedirectToAction("Login", "Admin");

    // 2. 调 Service 取消
    var result = await _reservationService.CancelByAdminAsync(id);

    TempData["Message"] = result.Success ? "预约已取消" : result.ErrorMessage;
    return RedirectToAction("Reservations");
}
```

### 3.3 Service 职责（ReservationService）

**`CancelByAdminAsync(int reservationId)`**

```
入口参数：reservationId

步骤 1 — 查预约记录：
  - 查 Reservations 表：WHERE Id == reservationId
  - 若不存在 → 返回失败 "预约记录不存在"

步骤 2 — 状态检查：
  - 若 Status == "已取消" → 返回失败 "该预约已经被取消"

步骤 3 — 执行取消：
  - UPDATE Reservations
    SET Status = "已取消", CancelledAt = @now
    WHERE Id == reservationId AND Status == "已预约"
  - 检查受影响行数，若为 0 → 返回失败 "取消失败，预约状态已变更"

步骤 4 — 返回成功：
  - 返回 CancelResult { Success = true }
```

> **管理员取消与用户取消的区别**：管理员取消不做"预约所属人"检查（管理员有权取消任意预约），用户取消需检查 `reservation.UserId == currentUserId`。

### 3.4 DataAccess 动作

| 步骤 | EF Core 操作 | 对应 SQL | 表 |
|------|-------------|----------|-----|
| 查预约 | `_context.Reservations.FindAsync(id)` | `SELECT * FROM Reservations WHERE Id = @id` | Reservations |
| 取消更新 | `entity.Status = "已取消"; entity.CancelledAt = now; _context.SaveChangesAsync()` | `UPDATE Reservations SET Status = '已取消', CancelledAt = @now WHERE Id = @id AND Status = '已预约'` | Reservations |

### 3.5 数据库变化

| 操作 | 表 | 影响 |
|------|-----|------|
| 查预约 | Reservations | 只读 |
| 更新状态 | Reservations | **修改一行**：Status = '已取消'，CancelledAt = 当前时间 |

### 3.6 状态流转

```
[预约记录存在, Status = "已预约"]
        │
        ▼
[管理员点击"取消预约"] ──→ Status = "已取消", CancelledAt = 当前时间
```

### 3.7 异常与边界情况

| 边界场景 | 检测层 | 响应 |
|----------|--------|------|
| 未登录直接访问 P07 | Controller Action 入口 Session 检查 | 重定向到 P06 `/Admin/Login` |
| 预约记录不存在 | Service 步骤 1 | P07 列表刷新，提示"预约记录不存在" |
| 预约已取消 | Service 步骤 2 | P07 列表刷新，提示"该预约已被取消" |
| POST 请求缺少 AntiForgeryToken | ASP.NET Core 框架 | 400 Bad Request |
| Session 过期后操作 | Controller Session 检查 | 重定向到 P06 |

## 4. 用户自行取消（P05 内）

用户端 P05 "我的预约"页的取消流程与链路二逻辑相同，差异仅在鉴权检查：

| 对比维度 | 用户取消 | 管理员取消 |
|----------|----------|-----------|
| 入口 | P05 我的预约页 | P07 预约管理页 |
| Controller | `ReservationController.Cancel(int id)` | `AdminManageController.CancelReservation(int id)` |
| 鉴权 | 检查 `reservation.UserId == currentUserId` | 不检查所属人，仅检查 Session |
| Service 方法 | `CancelByUserAsync(userId, reservationId)` | `CancelByAdminAsync(reservationId)` |
| 成功后跳转 | P05 本页（列表刷新） | P07 本页（列表刷新） |

## 5. 后续代码实现衔接点

1. **Entity 类创建**：按 08-数据库设计的表结构创建 C# Entity 类（`Seat.cs`、`Reservation.cs` 等）。
2. **AppDbContext 配置**：在 `OnModelCreating` 中用 Fluent API 配置外键、唯一索引（`HasIndex` + `HasFilter`）。
3. **Service 接口与实现**：先定义 `ISeatService` / `IReservationService` / `IAdminService` 接口，再实现具体类。
4. **Controller → View 数据流**：每个页面按 ViewModel 的字段填充 ViewData / Model，保持与原型 v1 的假数据格式一致。
5. **Session 注册**：`Program.cs` 中 `builder.Services.AddSession()` + `app.UseSession()`。
6. **种子数据**：`DbInitializer.Seed()` 方法中批量插入 Users / Admins / SeatAreas / Seats 种子数据，并可选插入示例 Reservations。

## 6. 默认补足链路 / 当前假设

前序文档已完整覆盖链路一和链路二的需求与流程，本轮无拆解级默认补足。仅补充以下实现级细节假设：

| 补足项 | 假设 | 理由 |
|--------|------|------|
| 事务包裹方式 | 使用 EF Core `Database.BeginTransactionAsync()` 包裹冲突检查 + 插入 | 08-SD 已强调"应用层 + 数据库唯一索引"双重保障，事务是应用层保障的前提 |
| 并发冲突提示文案 | "预约失败，该时段刚刚被抢走" | 课堂项目面向学生，文案需要友好易懂 |
| 管理员取消不检查所属人 | 管理员有权取消任意用户的预约 | 04-页面树 P07 职责为"预约管理"而非"个人预约"，符合管理端设计 |
| `GetCurrentUserId()` 实现 | 从 QueryString 参数 `?uid=` 读取，或从隐藏域 `@Html.Hidden("currentUserId")` 读取 | 07-SD 明确"用户端不存 Session"，体验账号切换通过页面参数传递 |
