# 图书馆座位预约系统 — 数据库设计

## 1. 数据设计目标

1. 支撑用户端"浏览座位 → 预约 → 查看/取消"完整闭环的数据读写。
2. 支撑管理端"预约筛选 → 座位管理 → 统计概览"的查询与管理操作。
3. 通过数据库约束保证：**同一座位 + 同一日期 + 同一时段 最多只有一条"已预约"状态记录**。
4. 字段命名、实体名称与前序文档（03-PRD-Lite / 04-页面树与业务流程 / 07-系统设计说明）保持一致。
5. 适合 SQL Server LocalDB + EF Core Code First，表结构简洁、关系清晰，适合单人课堂项目。

---

## 2. 核心实体清单

前序文档（07-系统设计说明 §4.5）已明确 5 个实体：

| 实体 | 表名 | 说明 | 来源 |
|------|------|------|------|
| 用户（体验） | `Users` | 体验账号，通过下拉框切换 | 03-PRD §5 / 07-SD §4.5 |
| 管理员 | `Admins` | 管理端登录账号（轻量，非 Identity） | 07-SD §4.5 |
| 座位区域 | `SeatAreas` | 图书馆内划分的自习区域 | 03-PRD §7 / 07-SD §4.5 |
| 座位 | `Seats` | 单个自习座位 | 03-PRD §7 / 07-SD §4.5 |
| 预约记录 | `Reservations` | 用户预约的核心记录 | 03-PRD §5 / 07-SD §4.5 |

**关于 TimeSlot（时段）**：前序文档将时段描述为预定义固定值（每日 7 个，如 08:00-10:00），未要求独立时段表。本设计将时段以 `VARCHAR` 字段存储于 `Reservations` 表中（如 `"08:00-10:00"`），不建独立表，减少 JOIN 复杂度。

**关于 ReservationLog（操作日志）**：前序文档未明确要求操作审计日志。本期通过 `Reservations.Status` 和 `CancelledAt` 字段追踪取消操作，不单独建 `ReservationLogs` 表。

## 3. ER 图（关系说明）

```
┌──────────┐       ┌──────────────┐       ┌──────────────┐
│  Admins  │       │  SeatAreas   │       │    Users     │
│──────────│       │──────────────│       │──────────────│
│ Id (PK)  │       │ Id (PK)      │       │ Id (PK)      │
│ Username │       │ Name         │       │ StudentId    │
│ Password │       │ Floor        │       │ DisplayName  │
│ CreatedAt│       │ SortOrder    │       │              │
└──────────┘       └──────┬───────┘       └──────┬───────┘
                          │ 1 : N               │ 1 : N
                          │                     │
                   ┌──────▼───────┐             │
                   │    Seats     │             │
                   │──────────────│             │
                   │ Id (PK)      │             │
                   │ SeatNumber   │             │
                   │ AreaId (FK)  │◄────────────┼──── FK to SeatAreas
                   │ Status       │             │
                   │ HasPower     │             │
                   │ HasLight     │             │
                   │ Description  │             │
                   └──────┬───────┘             │
                          │ 1 : N               │
                          │                     │
                   ┌──────▼─────────────────────▼─────┐
                   │         Reservations              │
                   │───────────────────────────────────│
                   │ Id (PK)                           │
                   │ UserId (FK → Users.Id)            │
                   │ SeatId (FK → Seats.Id)            │
                   │ ReservationDate (DATE)            │
                   │ TimeSlot (VARCHAR)                │
                   │ Status (VARCHAR)                  │
                   │ CreatedAt                         │
                   │ CancelledAt                       │
                   └───────────────────────────────────┘
```

**关系摘要**：

| 关系 | 类型 | 说明 |
|------|------|------|
| SeatAreas → Seats | 1 : N | 一个区域包含多个座位 |
| Users → Reservations | 1 : N | 一个用户可以有多条预约 |
| Seats → Reservations | 1 : N | 一个座位可以产生多条预约（不同日期/时段） |
| Admins | 独立 | 管理员表与预约/座位无直接外键关系，通过应用层鉴权 |

**预约唯一性约束（核心）**：
`(SeatId, ReservationDate, TimeSlot)` 组合 + `Status = '已预约'` → 最多一条。通过唯一索引实现（见 §9）。

## 4. 表结构设计

### 4.1 Users（用户表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `Id` | INT | PK, IDENTITY | 自增主键 |
| `StudentId` | NVARCHAR(20) | NOT NULL, UNIQUE | 学号（用作体验账号标识） |
| `DisplayName` | NVARCHAR(50) | NOT NULL | 显示名称（如"张三"） |

**种子数据**（3 个体验账号）：

| StudentId | DisplayName |
|-----------|-------------|
| 2024001 | 张三 |
| 2024002 | 李四 |
| 2024003 | 王五 |

### 4.2 Admins（管理员表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `Id` | INT | PK, IDENTITY | 自增主键 |
| `Username` | NVARCHAR(50) | NOT NULL, UNIQUE | 登录用户名 |
| `PasswordHash` | NVARCHAR(256) | NOT NULL | 密码哈希（SHA256 或 BCrypt） |

**种子数据**（1 个管理员）：

| Username | PasswordHash（示例明文 → 哈希） |
|----------|-------------------------------|
| admin | SHA256("admin123") |

> 注：课堂项目可直接存储 SHA256 哈希，不强制 BCrypt，降低实现复杂度。

### 4.3 SeatAreas（座位区域表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `Id` | INT | PK, IDENTITY | 自增主键 |
| `Name` | NVARCHAR(50) | NOT NULL | 区域名称（如"A区 自习室"） |
| `Floor` | INT | NOT NULL | 楼层（如 3） |
| `SortOrder` | INT | NOT NULL, DEFAULT 0 | 排序权重，越小越靠前 |

**种子数据**：

| Name | Floor | SortOrder |
|------|-------|-----------|
| A区 自习室 | 3 | 1 |
| B区 阅览区 | 4 | 2 |

### 4.4 Seats（座位表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `Id` | INT | PK, IDENTITY | 自增主键 |
| `SeatNumber` | NVARCHAR(20) | NOT NULL | 座位编号（如"A-01"） |
| `AreaId` | INT | FK → SeatAreas.Id, NOT NULL | 所属区域 |
| `Status` | NVARCHAR(20) | NOT NULL, DEFAULT '可用' | 座位物理状态：可用 / 停用 |
| `HasPower` | BIT | NOT NULL, DEFAULT 0 | 是否有电源插座 |
| `HasLight` | BIT | NOT NULL, DEFAULT 0 | 是否有台灯 |
| `Description` | NVARCHAR(200) | NULL | 备注（如"靠窗双人桌"） |

**种子数据**（对应原型 v1 假数据）：

| SeatNumber | AreaId | Status | HasPower | HasLight |
|------------|--------|--------|----------|----------|
| A-01 | 1 | 可用 | 1 | 1 |
| A-02 | 1 | 可用 | 1 | 0 |
| A-03 | 1 | 可用 | 0 | 1 |
| A-04 | 1 | 停用 | 0 | 0 |
| B-01 | 2 | 可用 | 1 | 1 |
| B-02 | 2 | 可用 | 0 | 1 |
| B-03 | 2 | 可用 | 1 | 0 |

### 4.5 Reservations（预约记录表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| `Id` | INT | PK, IDENTITY | 自增主键 |
| `UserId` | INT | FK → Users.Id, NOT NULL | 预约用户 |
| `SeatId` | INT | FK → Seats.Id, NOT NULL | 预约座位 |
| `ReservationDate` | DATE | NOT NULL | 预约日期（如 2026-07-10） |
| `TimeSlot` | NVARCHAR(20) | NOT NULL | 时段（如"08:00-10:00"） |
| `Status` | NVARCHAR(20) | NOT NULL, DEFAULT '已预约' | 预约状态：已预约 / 已取消 |
| `CreatedAt` | DATETIME2 | NOT NULL, DEFAULT GETDATE() | 创建时间 |
| `CancelledAt` | DATETIME2 | NULL | 取消时间（取消时写入） |

**预定义时段（7 个）**：

| 编号 | TimeSlot 值 |
|------|-------------|
| 1 | 08:00-10:00 |
| 2 | 10:00-12:00 |
| 3 | 12:00-14:00 |
| 4 | 14:00-16:00 |
| 5 | 16:00-18:00 |
| 6 | 18:00-20:00 |
| 7 | 20:00-22:00 |

## 5. 关键字段说明

| 字段 | 表 | 说明 |
|------|-----|------|
| `Seats.Status` | Seats | **物理状态**（可用/停用），由管理员控制，与预约无关。停用的座位在用户端不出现、不可约。 |
| `Reservations.Status` | Reservations | **预约状态**（已预约/已取消）。仅"已预约"参与冲突检查。 |
| `Reservations.ReservationDate` | Reservations | DATE 类型，不含时间部分。与 `TimeSlot` 组合确定预约时间段。 |
| `Reservations.TimeSlot` | Reservations | 存储预定义时段字符串。与原型 v1 的时段列表保持一致。 |
| `Reservations.CancelledAt` | Reservations | 记录取消时间。非 NULL 时 `Status = '已取消'`。用于统计当日取消数。 |
| `Seats.HasPower` / `Seats.HasLight` | Seats | 设施标识，在 P03 座位详情页展示。 |

## 6. 状态字段设计

### 6.1 座位物理状态（Seats.Status）

| 值 | 含义 | 用户端表现 | 管理员操作 |
|----|------|-----------|-----------|
| `可用` | 正常可用 | 出现在列表，标记"可预约" | 可点击"停用" |
| `停用` | 维护/故障 | 不出现或标记"已停用"（灰色，不可点） | 可点击"启用" |

### 6.2 预约状态（Reservations.Status）

| 值 | 含义 | 触发条件 |
|----|------|----------|
| `已预约` | 有效预约 | 用户提交成功（默认值） |
| `已取消` | 用户主动取消 | 用户在 P05 点击"取消"并确认 |

**状态流转**：

```
[提交预约] → 已预约 → [用户取消] → 已取消
```

> 本期不实现"已签到""已超时""已完成"等状态，仅保留最简单的二态流转。

## 7. 查询与筛选字段设计

### 7.1 用户端查询

| 查询需求 | 对应 SQL / LINQ | 说明 |
|----------|----------------|------|
| 按区域列出可用座位 | `Seats.Where(s => s.AreaId == areaId && s.Status == "可用")` | P02 座位列表 |
| 查看某座位指定日期时段是否可约 | `!Reservations.Any(r => r.SeatId == seatId && r.ReservationDate == date && r.TimeSlot == slot && r.Status == "已预约")` | P04 冲突检查 |
| 查看我的预约 | `Reservations.Where(r => r.UserId == userId).OrderByDescending(r => r.CreatedAt)` | P05 我的预约 |
| 查看某座位详情 + 今日时段占用 | `Reservations.Where(r => r.SeatId == id && r.ReservationDate == today && r.Status == "已预约")` | P03 详情页时段状态 |

### 7.2 管理端查询

| 查询需求 | 对应筛选方式 | 说明 |
|----------|-------------|------|
| 按日期筛选预约 | `ReservationDate = @date` | P07 日期参数 |
| 按状态筛选预约 | `Status = @status` | P07 状态下拉 |
| 按区域筛选预约 | `Seat.AreaId = @areaId`（JOIN Seats） | P07 区域下拉 |
| 按用户筛选 | `UserId = @userId` | P07 用户搜索 |
| 今日预约总数 | `COUNT(*) WHERE ReservationDate = today AND Status = '已预约'` | P09 统计卡片 |
| 各区域预约数 | `GROUP BY AreaId, ReservationDate` | P09 区域分布表 |

### 7.3 索引建议

```sql
-- 预约冲突检查（核心索引，覆盖最频繁的查询）
CREATE UNIQUE INDEX IX_Reservations_Conflict 
    ON Reservations (SeatId, ReservationDate, TimeSlot) 
    WHERE Status = '已预约';

-- 管理端筛选查询
CREATE INDEX IX_Reservations_Date_Status 
    ON Reservations (ReservationDate, Status);

-- 用户个人预约列表
CREATE INDEX IX_Reservations_UserId 
    ON Reservations (UserId) 
    INCLUDE (SeatId, ReservationDate, TimeSlot, Status, CreatedAt);
```

## 8. 统计字段与统计口径

| 统计项 | 口径 | 对应页面 |
|--------|------|----------|
| 座位总数 | `SELECT COUNT(*) FROM Seats WHERE Status = '可用'` | P09 统计卡片 |
| 今日预约数 | `SELECT COUNT(*) FROM Reservations WHERE ReservationDate = @today AND Status = '已预约'` | P09 统计卡片 |
| 今日取消数 | `SELECT COUNT(*) FROM Reservations WHERE ReservationDate = @today AND Status = '已取消'` | P09 统计卡片 |
| 各区域预约分布 | `SELECT a.Name, COUNT(r.Id) FROM SeatAreas a LEFT JOIN Seats s ON a.Id = s.AreaId LEFT JOIN Reservations r ON s.Id = r.SeatId AND r.ReservationDate = @today AND r.Status = '已预约' GROUP BY a.Name` | P09 区域分布表 |

> 统计为实时查询，不建物化视图或缓存表，课堂项目数据量小，实时 COUNT 足够。

## 9. 数据一致性与约束说明

| 约束类型 | 实现方式 | 说明 |
|----------|----------|------|
| **预约冲突（核心）** | 数据库唯一索引 `IX_Reservations_Conflict`（WHERE Status = '已预约'） | 同一座位+日期+时段最多一条有效预约。即使应用层漏检，数据库也会拒绝插入。 |
| **用户外键** | `FK_Reservations_Users` | 预约必须关联有效用户 |
| **座位外键** | `FK_Reservations_Seats` | 预约必须关联有效座位 |
| **区域外键** | `FK_Seats_SeatAreas` | 座位必须属于有效区域 |
| **状态值校验** | 应用层枚举 / 常量 | 前端下拉 + 后端枚举约束 Seats.Status ∈ {可用, 停用}，Reservations.Status ∈ {已预约, 已取消} |
| **取消幂等** | 应用层检查 `Status == '已预约'` 才允许取消 | 防止重复取消 |
| **停用座位不可约** | 应用层在冲突检查前先查 `Seats.Status == '可用'` | 停用座位拒绝预约 |

## 10. 当前阶段实现边界

| 边界项 | 本期实现 | 本期不实现 |
|--------|----------|------------|
| 时段 | 7 个固定时段字符串，硬编码在应用层常量中 | 动态时段表 / 自定义时长 |
| 用户表 | 3 个种子体验用户 | 注册 / 修改密码 / 找回密码 |
| 管理员表 | 1 个种子管理员（admin/admin123） | 管理员列表 / 多管理员 |
| 操作日志 | 通过 Reservations.CancelledAt 追踪取消 | 独立日志表 / 完整审计 |
| 签到 | 不实现 | - |
| 软删除 | 不实现（取消操作只改 Status，不删记录） | - |

## 11. 默认补足项 / 当前假设

以下内容为前序文档未明确而本阶段补足的设计决策：

| 补足项 | 假设 | 理由 |
|--------|------|------|
| `Admins` 表独立存在 | 管理员账号与体验用户分离，管理员不参与预约 | 07-SD 明确 Admin 为独立实体 |
| 时段存储为 VARCHAR 而非独立 TimeSlots 表 | 7 个固定时段直接以字符串形式存入 Reservations | 03-PRD 未要求动态时段管理，降低复杂度 |
| `ReservationLog` 不单独建表 | 取消操作通过 Reservations 表自身的 Status + CancelledAt 追踪 | 03-PRD / 04 / 07 均未提及操作审计需求 |
| `PasswordHash` 使用 SHA256 | 课堂项目不引入 BCrypt 依赖 | 07-SD 建议轻量实现，学生可理解 |
| 索引使用过滤唯一索引（Filtered Unique Index） | `WHERE Status = '已预约'` 过滤索引仅在 SQL Server 中支持 | SQL Server LocalDB 原生支持过滤索引 |
