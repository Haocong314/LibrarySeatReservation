# 图书馆座位预约系统 — 项目立项单

## 1. 项目基本信息

| 项目 | 内容 |
|------|------|
| 项目名称 | 图书馆座位预约系统 |
| 项目代号 | LibSeat |
| 项目类型 | 课堂单人开发项目（课程第三阶段） |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core + Bootstrap |
| 开发人员 | 软件专业实习生（个人） |
| 提交方式 | GitHub / Gitee 仓库链接 |
| 项目周期 | 约 4-6 周（课余时间推进） |

## 2. 项目背景

大学图书馆每到考试周和自习高峰时段，经常出现"一座难求"的情况。同学们要么早起排队抢座，要么到了图书馆发现没有空位白跑一趟。目前很多学校还没有一个轻量的在线预约工具，或者已有系统太重、太卡。本项目就是一个简化版的座位预约系统，模拟图书馆管理员和学生的日常使用场景，作为课程阶段的个人实战项目。

## 3. 目标用户

- **学生用户**：需要提前查看图书馆座位情况、预约座位的在校学生。
- **管理员**：需要管理座位、查看预约情况、处理异常预约的图书馆工作人员。

## 4. 项目目标

做一个"能跑、能演示、能讲清楚"的座位预约小系统，达成三个核心目标：

1. 学生可以在手机或电脑上浏览图书馆座位、提交预约、查看自己的预约、取消预约。
2. 管理员可以在后台管理座位信息、查看全部预约、做一些简单的统计（比如今天预约了多少人）。
3. 整个系统代码结构清晰，数据库设计合理，可以作为结课作业或简历作品展示。

## 5. 本期必须完成内容

### 5.1 用户端（5 页）

| 页面 | 功能说明 |
|------|----------|
| 用户首页 | 展示图书馆基本信息、快捷入口（找座位、我的预约） |
| 座位列表页 | 按区域分类展示座位，显示每个座位是否可预约 |
| 座位详情页 | 展示单个座位的详细信息（位置、设施等）+ 预约入口 |
| 预约提交页 | 选择日期和时段，确认并提交预约 |
| 我的预约页 | 列出当前用户全部预约，支持取消预约操作 |

### 5.2 管理端（4 页）

| 页面 | 功能说明 |
|------|----------|
| 管理员登录页 | 管理员账号密码登录入口 |
| 预约管理页 | 查看所有预约记录，可按日期/状态筛选 |
| 座位管理页 | 增加、修改、停用/启用座位 |
| 统计页 | 展示今日预约总数、各区域预约分布等简单统计 |

### 5.3 后端核心

- Entity Framework Core + SQL Server LocalDB 做数据持久化
- 预约冲突检查：同一座位同一日期同一时段不能重复预约
- 轻量登录：用户端用体验账号切换，管理端用 Session/Cookie

## 6. 本期暂不开发内容

- 支付功能（不涉及任何费用）
- 扫码签到/签退
- 短信或微信消息推送
- 复杂账号体系（注册、邮箱验证、找回密码）
- 多馆区支持（只做单图书馆）
- 实时座位动态图（不做 Floor Map）
- 移动端 App

## 7. 页面规模与页面分布

共计 **9 个核心页面**：

| 端 | 页面数 | 页面列表 |
|----|--------|----------|
| 用户端 | 5 | 用户首页、座位列表页、座位详情页、预约提交页、我的预约页 |
| 管理端 | 4 | 管理员登录页、预约管理页、座位管理页、统计页 |

这个规模对单人课堂项目来说刚好：太少演示不出东西，太多根本做不完。

## 8. 关键业务主流程

### 8.1 用户预约主流程

```
用户打开首页 → 点击"找座位" → 浏览座位列表 → 点某个座位看详情
→ 选择日期和时段 → 提交预约 → 预约成功 → 在我的预约中查看
→ 如需取消，在"我的预约"中点击取消 → 取消成功
```

### 8.2 管理员管理主流程

```
管理员打开后台 → 输入账号密码登录 → 进入预约管理
→ 按日期/状态筛选预约 → 查看预约详情
→ 进入座位管理 → 添加新座位或停用旧座位
→ 进入统计页 → 查看今日预约情况
```

## 9. 成功标准

以下标准必须是"打开浏览器就能当场演示"的动作：

| 编号 | 成功标准 | 演示方式 |
|------|----------|----------|
| S1 | 学生能浏览座位列表，看到每个座位是否可约 | 打开座位列表页，展示区域 + 状态 |
| S2 | 学生能选择日期、时段，成功提交预约 | 从详情页进预约提交页，填完表单点提交，跳转到"我的预约"看到新记录 |
| S3 | 学生能取消自己已提交的预约 | 在"我的预约"页点击"取消"，页面刷新后该预约变为已取消 |
| S4 | 同一座位同一日期同一时段不能重复预约 | 用另一个体验账号尝试预约已占时段，系统提示"该时段已被预约" |
| S5 | 管理员登录后能查看全部预约并筛选 | 登录后台，在预约管理页按日期筛选，看到正确的预约列表 |
| S6 | 管理员能添加新座位、停用旧座位 | 在座位管理页添加座位→座位列表出现新座位；停用某座位→用户端该座位显示不可约 |
| S7 | 统计页能正确显示今日预约总数 | 打开统计页，今日预约数 = 预约管理页中今日的预约条数 |

## 10. 风险与假设

| 风险 | 说明 | 应对 |
|------|------|------|
| 时间不够 | 课余时间碎片化，每周能投入时间不稳定 | 严格控制 9 页不扩展；优先保证用户主链路跑通 |
| 页面失控 | 对美工和前端投入过多，后端没时间做 | 用 Bootstrap 默认风格，不追求个性化 UI |
| 数据库过重 | 表设计太复杂，迁移和种子数据出问题 | 前期只做最核心 5-6 张表，不要一口气建十几张 |
| AI 代码不稳定 | AI 生成的代码可能有 Bug 或跟需求不一致 | 每个 Sprint 完成后必须跑通主流程测试 |
| EF Core 踩坑 | 新手对 Migration、DbContext 不够熟悉 | 优先用简单的一对多关系，避免多对多和复杂 LINQ |
| 电脑环境问题 | LocalDB 没装上、SDK 版本不对等 | 开工前先确保 `dotnet new` 和 `dotnet run` 能跑通 |

**假设**：开发环境和运行时都在 Windows 上，已安装 .NET 8 SDK 和 SQL Server LocalDB。
